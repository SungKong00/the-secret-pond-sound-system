〈비밀의 연못〉 녹음·루프 자동화 시스템 개발 요청서

3줄 요약

1. 관객이 버튼이나 터치 구조물에 손을 올리고 있는 동안만 음성이 녹음되고, 손을 떼면 자동으로 녹음이 종료되는 시스템이 필요합니다.
2. 녹음된 음성은 간단한 EQ, 필터, 볼륨, 리버브 정도의 처리를 거친 뒤 기존 루프 사운드 위에 자동으로 겹쳐집니다.
3. 사운드 배스 공간에서는 계속 반복 재생되는 루프가 있고, 새 녹음이 들어올 때마다 그 루프 위에 목소리 레이어가 하나씩 쌓여야 합니다.


1. 개발 목표

이 시스템은 관객이 녹음실에서 자신의 말을 남기면, 그 소리가 자동으로 처리되어 사운드 배스 공간의 반복 루프에 추가되는 구조입니다.

복잡한 음성 분석이나 AI 변형은 필요하지 않습니다.
핵심은 다음 네 가지입니다.

녹음하기 → 소리 톤 정리하기 → 기존 루프에 겹치기 → 계속 재생하기

즉, 개발자가 만들어야 하는 것은 자동 녹음 및 루프 누적 시스템입니다.


2. 전체 동작 흐름

기본 흐름

1. 사운드 배스에는 기본 루프 사운드가 계속 재생되고 있다.
2. 관객이 녹음실에 들어간다.
3. 관객이 버튼이나 터치 구조물에 손을 올린다.
4. 손을 올리고 있는 동안만 녹음된다.
5. 손을 떼면 녹음이 종료된다.
6. 녹음된 파일에 EQ, 필터, 리버브, 볼륨 조절 등을 적용한다.
7. 처리된 음성을 기존 루프 위에 겹쳐 새 루프 파일을 만든다.
8. 사운드 배스 재생 시스템이 새 루프를 자동으로 반영한다.
9. 이후 다른 관객이 녹음하면 같은 방식으로 계속 쌓인다.


3. 개발해야 하는 기능

1) 녹음 기능

관객이 특정 구조물에 손을 올리면 녹음이 시작되고, 손을 떼면 녹음이 끝나야 합니다.

필요 기능:
- 버튼 또는 터치 센서 입력 감지
- 입력이 ON일 때만 녹음
- 입력이 OFF되면 녹음 종료
- 너무 짧은 녹음은 무시
- 최대 녹음 시간 설정
- 녹음 중 상태 표시

권장값:
- 최소 녹음 시간: 3초
- 최대 녹음 시간: 60초
- 저장 포맷: WAV
- 샘플레이트: 44.1kHz 또는 48kHz
- 비트 depth: 16bit 또는 24bit


2) 간단한 사운드 처리 기능

녹음된 음성은 복잡하게 변형할 필요는 없습니다.
다만 사운드 배스의 전체 노이즈 루프와 잘 섞이도록 톤을 정리해야 합니다.

필요한 처리:
- 볼륨 노멀라이즈
- 하이패스 필터
- 로우패스 필터
- EQ 밴드 조절
- 리버브
- 약한 딜레이 선택 가능
- 페이드 인 / 페이드 아웃
- 전체 루프와 섞일 때 너무 튀지 않도록 볼륨 조절

예시 처리 방향:
- 말소리가 너무 선명하게 튀지 않게 중고역 일부 정리
- 저역 노이즈 제거
- 리버브로 공간감 추가
- 전체 루프 안에 자연스럽게 묻히도록 볼륨 낮추기
- 여러 명의 목소리가 쌓였을 때 화이트 노이즈처럼 느껴지도록 밸런스 조절


3) 루프 합성 기능

기존에 재생 중인 루프 파일 위에 새로 녹음된 음성을 겹쳐야 합니다.

필요 기능:
- 현재 루프 파일 불러오기
- 새 녹음 파일 불러오기
- 새 녹음을 루프 안의 특정 위치에 배치
- 기존 루프와 새 녹음 레이어를 믹싱
- 새 루프 파일 생성
- 기존 루프를 새 루프로 교체

루프 길이 권장:
- 프로토타입: 3분 또는 5분
- 전시 버전: 5분 이상 가능

새 녹음 배치 방식:
- 루프 안의 랜덤 위치에 삽입
- 또는 최근 녹음은 초반에 조금 더 잘 들리게 배치
- 시간이 지나며 계속 다른 목소리와 섞이는 구조


4) 재생 기능

사운드 배스에서는 루프가 끊기지 않고 계속 재생되어야 합니다.

필요 기능:
- 시스템 실행 시 자동 재생
- 현재 루프 반복 재생
- 새 루프 파일이 생성되면 자동 반영
- 파일 교체 시 끊김 최소화
- 가능하면 크로스페이드 적용
- 오류 시 기존 루프 계속 재생

간단한 방식:
지금 재생 중인 루프가 끝나면 새로 생성된 루프 파일로 교체

조금 더 좋은 방식:
새 루프가 만들어지면 기존 루프와 새 루프를 크로스페이드하며 자연스럽게 전환


5) 참여자 수 기록

작품 운영을 위해 몇 명이 녹음했는지만 기록하면 됩니다.

필요 기능:
- 녹음이 정상 완료되면 참여자 수 +1
- 너무 짧거나 실패한 녹음은 카운트하지 않음
- 현재 참여자 수를 운영자 화면에서 확인 가능

남길 정보:
- 참여자 수
- 녹음 시간
- 처리 성공 여부
- 시스템 오류 로그

개인정보나 녹음 내용 기록은 필요 없습니다.


4. 파일 구조

개발자는 다음처럼 단순한 폴더 구조로 만들면 됩니다.

/project
  /recordings_temp
    방금 녹음된 임시 음성 파일

  /processed
    EQ/필터 처리된 음성 파일

  /loops
    pond_loop_current.wav
    pond_loop_previous.wav

  /logs
    system_log.txt
    participant_count.json

단순 버전에서는 최종적으로 pond_loop_current.wav만 계속 갱신되면 됩니다.


5. 권장 시스템 구조

가장 단순한 구조

버튼/터치 센서
→ 컴퓨터
→ 녹음
→ EQ/필터/리버브 처리
→ 기존 루프와 믹싱
→ 새 루프 저장
→ 사운드 배스에서 재생

권장 개발 방식

초기 프로토타입은 Python 기반으로 충분합니다.

사용 가능 기술:
- Python
- sounddevice 또는 pyaudio: 녹음
- pydub 또는 librosa: 오디오 처리
- ffmpeg: 파일 변환
- watchdog: 새 파일 감지
- simpleaudio / pygame / Max/MSP 연동: 재생
- Arduino 또는 ESP32: 버튼/터치 센서 입력

사운드 실험이 중요하면 Max/MSP를 함께 써도 좋습니다.

추천 조합:
- Python: 녹음, 파일 생성, 루프 합성, 파일 관리
- Max/MSP: 사운드 재생, EQ, 다채널 출력, 실시간 조정
- Arduino/ESP32: 손 접촉 센서 입력


6. 개발 단계

1단계: 기본 MVP

가장 먼저 이것만 되면 됩니다.

- 버튼을 누르면 녹음 시작
- 버튼을 떼면 녹음 종료
- 녹음 파일 저장
- 간단한 EQ/필터 적용
- 기존 루프와 겹치기
- 새 루프 파일 생성
- 새 루프 재생

2단계: 전시용 안정화

- 터치 센서 연결
- 최대 녹음 시간 제한
- 너무 짧은 녹음 무시
- 녹음 중 LED 표시
- 처리 중 LED 표시
- 참여자 수 카운트
- 오류 발생 시 기존 루프 유지

3단계: 사운드 배스 출력 확장

- 스테레오 출력
- 근거리 스피커 출력
- 베이스 셰이커 출력
- 익사이터 출력
- 채널별 볼륨 조절
- 앉았을 때 더 강하게 느껴지는 출력 구조 테스트


7. 개발자에게 요청할 최종 결과물

개발자에게 요청할 것은 다음입니다.

1. 버튼 또는 터치 센서 기반 녹음 시작/종료 시스템
2. 녹음 파일 자동 생성 기능
3. 녹음 파일에 EQ/필터/리버브를 적용하는 처리 기능
4. 처리된 음성을 기존 루프 위에 자동으로 겹치는 기능
5. 새 루프 파일을 자동 생성하는 기능
6. 사운드 배스에서 루프를 계속 재생하는 기능
7. 새 루프가 생성되면 자동 반영하는 기능
8. 참여자 수 카운트 기능
9. 녹음 중 / 처리 중 / 완료 상태 표시 기능
10. 설치 및 실행 방법 문서


8. 개발자에게 보낼 짧은 요청문

〈비밀의 연못〉은 관객이 녹음실에서 버튼이나 터치 구조물에 손을 올리고 있는 동안만 자신의 말을 녹음하고, 손을 떼면 녹음이 끝나는 작품입니다. 녹음된 음성은 복잡한 AI 분석이나 비식별화가 필요한 것은 아니고, EQ 밴드 조절, 필터, 리버브, 볼륨 정리 정도를 거쳐 기존 사운드 루프 위에 계속 겹쳐지면 됩니다.

개발해야 할 것은 녹음 자동화, 센서 기반 녹음 시작/종료, 녹음 파일 생성, 간단한 사운드 처리, 기존 루프와의 자동 믹싱, 새 루프 파일 생성, 사운드 배스 재생, 새 루프 자동 반영, 참여자 수 기록입니다.

최종적으로는 여러 사람의 목소리가 하나의 루프 위에 겹겹이 쌓여, 개별 문장이 또렷하게 들리기보다는 목소리들이 누적된 노이즈처럼 들리는 구조를 만들고 싶습니다.


9. 2026-06-03 개발 방향 개정사항

이후 구현은 위 요청서를 기본으로 삼되, 실제 개발에서는 다음 내용을 우선 적용합니다.

1) 커밋 및 개발 방식

- 커밋 메시지는 "feat;한글 설명"처럼 세미콜론을 쓰는 한글 Conventional Commit 형식으로 작성합니다.
- 큰 단위 한 번에 커밋하지 않고, 기능이 검증되는 작은 단위로 나눠 커밋합니다.
- 각 기능 단위는 테스트를 먼저 작성하고, 구현 후 테스트와 린트를 확인합니다.
- 서브 에이전트가 계획과 구현을 비판적으로 검토하고, 타당한 지적은 반영한 뒤 다음 단계로 진행합니다.

2) 입력 방식

- MVP에서는 버튼/터치센서 대신 브라우저가 포커스를 가진 상태에서 스페이스바를 누르고 있는 동안 녹음합니다.
- 실수 녹음을 막기 위해 운영자가 먼저 Arm 상태로 전환해야 스페이스바 녹음이 작동합니다.
- 하드웨어 버튼, 터치센서, Arduino/ESP32 연동은 후속 단계로 둡니다.

3) 녹음 시간

- 최소 녹음 시간은 3초입니다.
- 최대 녹음 시간은 기존 권장값 60초가 아니라 120초로 설정합니다.

4) 목소리 스택 길이

- 목소리 합본의 기본 길이는 1분입니다.
- 목소리 스택은 1분 루프로 반복됩니다.
- 향후 운영자가 쉽게 바꿀 수 있도록 설정값으로 분리하되, 구현이 복잡해지면 기본 1분을 우선합니다.

5) 긴 WAV 파일 처리

- 1분을 넘는 목소리 WAV 파일은 1분 단위로 나눠 여러 스택 추가분으로 처리합니다.
- 마지막 조각이 1분보다 짧으면 그 짧은 조각도 1분 스택 안에서 루프되도록 처리합니다.

6) 테스트 동작과 실제 동작 분리

- 테스트/리허설 모드에서는 폴더 안의 목소리 WAV 파일들을 불러와 목소리 스택을 만들 수 있어야 합니다.
- 테스트/리허설 모드에서는 새 목소리 스택을 쌓을 때도 해당 폴더 안에 개별 목소리 WAV 파일을 남길 수 있습니다.
- 실제 운영 모드에서는 목소리 스택을 추가하더라도 원본 단일 목소리 WAV 파일은 남기지 않습니다.
- 실제 운영 모드에서는 녹음된 음성을 처리하고 스택에 합친 뒤, 개별 단일 목소리 파일은 삭제하거나 임시 파일로만 취급합니다.

7) 기존 스택에서 시작

- 전시 시작 시 목소리 스택이 비어 있을 수도 있고, 이미 어느 정도 쌓여 있는 기존 목소리 스택 WAV에서 시작할 수도 있어야 합니다.
- 기존 스택에서 시작할 때도 이후 새 녹음은 같은 1분 스택 규칙에 따라 누적됩니다.

8) UI 방향

- UI는 LazyWeb MCP를 참고해 구현합니다.
- 기본은 운영자가 실수 없이 볼 수 있는 대시보드입니다.
- 시각 분위기는 가능하면 몽환적, 사이키델릭, 사이버펑크 느낌을 사용합니다.
- 단, Arm/Disarm, 녹음 중, 처리 중, 오류 상태는 장식보다 명확성이 우선입니다.

9) 장치 선택과 적용 방식

- 운영자 대시보드에서 입력/출력 오디오 장치 후보를 확인하고 초안으로 선택할 수 있게 합니다.
- 장치 선택은 Mac/Windows에서 장치 이름과 ID가 달라질 수 있으므로, 앱 시작 시 현재 환경의 장치 목록을 다시 확인합니다.
- 현재 MVP에서는 입력/출력 장치 변경, 샘플레이트 변경, 채널 수 변경을 `Apply and Restart`로 즉시 반영하지 않습니다.
- 장치나 오디오 포맷을 바꾸려면 초안 또는 설정 파일을 바꾼 뒤 앱을 재시작하는 방식을 우선합니다.
- 앱 재시작 시 입력/출력 장치 또는 오디오 포맷 초안이 기존 활성 설정과 다르면, 그 초안을 시작 설정으로 승격합니다.
- EQ/볼륨/레이어 활성화 변경은 초안 저장 후 `Apply and Restart`로 렌더링 파일을 다시 만들고 재생기에 반영합니다.
- 출력 중에도 `Apply and Restart`를 사용할 수 있으며, 적용 과정에서 출력 스트림을 잠시 멈춘 뒤 새 렌더를 검증하고 다시 시작합니다.
- 적용 실패 시에는 가능한 범위에서 이전 렌더/플레이어/출력 상태로 복원하고, 입력/출력 장치나 오디오 포맷 변경은 계속 앱 재시작으로 처리합니다.

10) 상태 동기화

- 운영자 대시보드는 WebSocket으로 상태를 받아오되, 연결이 불가능하면 기존 HTTP polling으로 되돌아갑니다.
- WebSocket 연결이 끊길 때 활성 녹음이 있으면 `stop_recording`과 같은 방식으로 종료해, 충분히 긴 녹음은 정상 스택에 반영합니다.

11) 현재 운영자 UI와 문서화 반영

- Phase 10 운영자 UI는 첫 화면 대시보드로 구현하며, 별도 랜딩 페이지를 두지 않습니다.
- 하드웨어 센서 입력은 MVP 범위에서 제외하고, Arm 상태에서 브라우저 Spacebar hold-to-record를 사용합니다.
- 헤더에는 Arm/Disarm, 녹음 상태, 출력 상태, 현재 모드, 최근 이벤트, 장치 상태, 참여자 수, `Unsaved audio changes`를 표시합니다.
- 출력 제어는 `Start Output`, `Stop Output`, `Restart Output`으로 분리하며, `Restart Output`은 현재 로드된 재생을 처음부터 다시 시작하는 제어입니다.
- `Apply and Restart`는 EQ/볼륨/레이어/렌더링 대상 오디오 설정을 초안에서 활성 설정으로 적용하는 경로이며, 장치/샘플레이트/채널 변경은 앱 재시작 후 시작 설정 승격으로 처리합니다.
- Voice Treatment에는 Soft / Misty / Dense / Clearer Voice 프리셋을 제공해 비전문 운영자가 기술 슬라이더만 보지 않아도 되게 합니다.
- 운영자 안내 문서는 `docs/operator-guide.md`, 현장 점검표는 `docs/audio-setup-checklist.md`에 둡니다.
- `secret-pond doctor --json`은 현장 로그용 기계 판독 readiness 보고서를 출력하고, `secret-pond doctor --strict`는 준비 음원/장치/의존성/쓰기 권한 같은 기본 현장 준비 상태가 실패하면 비정상 종료합니다.
- 단, doctor strict는 마이크 권한, 실제 녹음/출력 스트림, 브라우저 동작, Windows 파일 교체까지 증명하지 않으므로 현장 점검표의 수동 확인을 유지합니다.
- `Reset Draft`는 기본 동작 버튼 줄에 노출하지 않고 `Maintenance` 안에 배치해, 초안 폐기 동작이 `Apply and Restart`와 같은 주요 운영 동작으로 오인되지 않게 합니다.
- 녹음 중에는 Reset Draft를 비활성화하고 API에서도 거부해, 녹음 중인 초안 상태가 우발적으로 폐기되지 않게 합니다.
